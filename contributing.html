💡 Ways to Contribute

There are many ways to contribute to Luminous Dynamics:

  • Report bugs - Help us identify and fix issues
  • Suggest features - Share your ideas for improvements
  • Write documentation - Improve guides, tutorials, and API docs
  • Fix bugs - Submit pull requests for known issues
  • Add features - Implement new functionality
  • Review code - Help review pull requests from other contributors
  • Improve tests - Add or enhance test coverage
  • Answer questions - Help other users on Discord or GitHub

🚀 Getting Started

Prerequisites

Before you begin, ensure you have the following installed:

  • Node.js 18.x or higher
  • npm 9.x or higher (or pnpm 8.x)
  • Git 2.x or higher
  • A code editor (we recommend VS Code)

Development Setup

Follow these steps to set up your development environment:

  1. Fork the repository

    Click the "Fork" button on GitHub to create your own copy

  2. Clone your fork
    git clone https://github.com/YOUR_USERNAME/luminousdynamics-io.git cd luminousdynamics-io
  3. Add upstream remote
    git remote add upstream https://github.com/Luminous-Dynamics/luminousdynamics-io.git
  4. Install dependencies
    npm install # or pnpm install
  5. Start development server
    npm run dev # or python -m http.server 8000
  6. Open in browser

    Navigate to http://localhost:8000

💡 Tip: We use a simple static site architecture. No build step is required for development—just open the HTML files in your browser or use a local server.

📝 Code Style Guidelines

JavaScript

  • Use ES6+ features (arrow functions, const/let, template literals)
  • Follow 2-space indentation
  • Use single quotes for strings
  • Add JSDoc comments for functions
  • Use meaningful variable names
  • Prefer const over let, avoid var
  • Use async/await for promises
/**
 * Calculate reading time for content
 * @param {HTMLElement} element - Content element
 * @returns {number} Reading time in minutes
 */
function calculateReadingTime(element) {
    const text = element.textContent;
    const words = text.trim().split(/\s+/).length;
    return Math.ceil(words / 225);
}

HTML

  • Use semantic HTML5 elements (header, nav, main, article, section)
  • Add ARIA labels for accessibility
  • Use 4-space indentation
  • Include alt text for all images
  • Validate with W3C validator

CSS

  • Use CSS custom properties for theming
  • Follow BEM naming for complex components
  • Mobile-first responsive design
  • Use rem/em for sizing (not px)
  • Add comments for sections
  • Support prefers-reduced-motion

Accessibility

  • WCAG AA compliance minimum (AAA preferred)
  • Keyboard navigation support
  • Screen reader compatibility
  • Color contrast ratio 4.5:1 minimum
  • Focus indicators on interactive elements

🌳 Git Workflow

Branching Strategy

We use a simple Git workflow:

  • main - Production-ready code
  • feature/feature-name - New features
  • fix/bug-name - Bug fixes
  • docs/topic-name - Documentation updates

Creating a Branch

# Update your local main git checkout main git pull upstream main # Create a new branch git checkout -b feature/amazing-feature

Commit Messages

Follow the Conventional Commits specification:

type(scope): subject body (optional) footer (optional)

Types:

  • feat - New feature
  • fix - Bug fix
  • docs - Documentation only
  • style - Formatting, missing semicolons, etc.
  • refactor - Code restructuring
  • perf - Performance improvement
  • test - Adding tests
  • chore - Maintenance tasks

Examples:

feat(api): add WebSocket support for real-time updates fix(ui): resolve mobile navigation overlay z-index issue docs(contributing): add code style guidelines

🔀 Pull Request Process

Before Submitting

  • Ensure your code follows our code style guidelines
  • Test your changes thoroughly
  • Update documentation if needed
  • Add or update tests for new functionality
  • Ensure all tests pass
  • Rebase on latest main branch

Submitting a Pull Request

  1. Push your branch
    git push origin feature/amazing-feature
  2. Open a pull request

    Go to GitHub and click "Compare & pull request"

  3. Fill out the PR template
    • Describe what you've changed and why
    • Reference related issues
    • Add screenshots for UI changes
    • List any breaking changes
  4. Wait for review

    Maintainers will review your PR and may request changes

  5. Address feedback

    Make requested changes and push new commits

  6. Merge!

    Once approved, a maintainer will merge your PR

⚠️ Important: Please do not force-push to your PR branch after opening a pull request. This makes it difficult to review changes.

🧪 Testing

Manual Testing

  • Test in multiple browsers (Chrome, Firefox, Safari, Edge)
  • Test on different devices (desktop, tablet, mobile)
  • Test with keyboard navigation
  • Test with screen reader (NVDA, JAWS, VoiceOver)
  • Test both themes (light and dark)
  • Test with reduced motion enabled

Automated Testing

Run automated tests before submitting:

# Lint code npm run lint # Run tests npm test # Check accessibility npm run a11y # Build for production npm run build
